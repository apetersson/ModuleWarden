# Analysis of the ModuleWarden v1 plan

## What's strong

The thesis is unusually disciplined. Most "AI security" pitches collapse the moment you ask what threat they actually catch; this one names a specific, growing class — the compromised-maintainer version bump (xz-utils, event-stream, ua-parser-js, the rspack/eslint-config-prettier style incidents) — and structures the entire product around it. The version-diff framing is not just rhetorical: it actually shrinks the input to a bounded, model-tractable artifact, which is the only way LLM review over package code becomes economically and accuracy-wise plausible.

The private-prompt argument as the defensible moat is the right answer to the obvious critique ("attackers can buy your scanner and tune around it"). The plan resists overclaiming it (TASK-1.1 explicitly excludes root admins from the secrecy boundary), which keeps the thesis honest.

Operational choices line up with the thesis. Approved-only metadata plus dist-tag rewriting (TASK-1.4) is a more thoughtful proxy behavior than most artifact gateways implement — flexible semver resolving to "newest approved" is what makes the gate non-disruptive in practice. Per-job disposable containers with a run-scoped RPC token (TASK-1.6) and zero credentials inside the sandbox is the correct shape. The Prisma + pg-boss + Postgres-only stance (TASK-1.16) is a good simplification: one operational boundary, one consistency story, no Redis sprawl. The replay-corpus evaluation bar (TASK-1.13) and "we will measure quarantine rate and escalations, not hand-wave" framing is exactly the discipline most of this category lacks.

## Real risks worth addressing now

**The cold-start case is the weakest case for your strongest argument.** TASK-1.5 #5 and TASK-1.9 #4 flag cold-start audits as "full-review/full-package" — but the diff thesis is what makes the LLM review tractable and your moat defensible. For a package with no predecessor, you're back to "audit a whole tarball," which is the regime where you've correctly said LLM review is weakest. This deserves its own explicit substrategy (e.g., much heavier weight on heuristic + provenance + maintainer reputation + first-pass scoping, with a stronger bias toward quarantine), not just "differently marked."

**Compute and latency aren't budgeted.** A real enterprise lockfile is 1.5–3k transitive deps, and npm publishes thousands of versions/day across popular packages. With a fresh Docker container per audit, plus possible escalation, plus re-audit campaigns when prompts/models/patterns change, you're looking at meaningful container-hours. The plan has no p95 audit-time SLO, no concurrency budget, no estimate of re-audit-campaign cost. "Proactive audit" without a budget is hope. I'd add an explicit non-functional acceptance criterion: first-time audit p95 (e.g., < 5 min), escalation p95 (e.g., < 20 min), and a re-audit throttling policy.

**The local OpenAI-compatible model assumption is load-bearing and under-specified.** The product depends on code-grade reasoning over real npm tarballs. Most "local OpenAI-compatible" deployments today (Ollama, vLLM, llama.cpp) don't run frontier-class models at frontier quality. The architecture should explicitly allow operators to point at a hosted endpoint if they accept the prompt-leakage tradeoff — and the threat model in TASK-1.1 should describe what the secrecy boundary degrades to in that case.

**Prompt secrecy boundary stops too early.** TASK-1.1 #4 excludes root infra admins but doesn't mention the model endpoint operator. If the local endpoint logs prompts (most do, by default), the moat leaks there. Worth a model-side policy note in operator docs (TASK-1.15) and ideally an acceptance criterion that the system warns when configured against an endpoint with unverified logging behavior.

**"Used graph" derived from lockfiles is fuzzier than it looks.** Monorepos have multiple lockfiles; dev-dependencies are excluded from production graphs but still installed in CI (and exploited at build time — that was the eslint-config-prettier vector); pnpm hoisting and yarn PnP produce different shapes. TASK-1.5 should make the scope of "used" explicit and probably configurable.

**Recorded-open egress is the right v1 choice, but the perimeter needs words.** If the audit container shares any network with internal infra, "no secrets worth exfiltrating" doesn't fully hold — lateral reachability is a separate concern. TASK-1.6 should specify the container's network is its own segment with no reachability to ModuleWarden internals besides the RPC bridge.

**Overrides don't auto-revoke.** TASK-1.10 lets security admins override a quarantine/block, but there's no flow for "new evidence after override invalidates the override." A re-audit-after-override hook in TASK-1.8 would close this.

## Smaller things worth tightening

The 16-task plan is well-sequenced but ships nothing incrementally — every task is required for "v1." A v0.1 internal milestone of "Verdaccio gating + lockfile import + manual allow/deny, no LLM yet" would let you exercise the proxy, dist-tag rewriting, and pg-boss plumbing under realistic load before the agent is anywhere near reliable. It would also produce real evidence-bundle artifacts to feed prompt iteration.

Scores are retained for "later calibration" (TASK-1.13 #4) but there's no plan for capturing labeled outcomes (admin override = correction signal, post-hoc incident reports = ground truth). Without that loop, the score corpus is just numbers. A small AC under TASK-1.13 or TASK-1.10 saying "overrides and post-hoc relabelings are first-class training signals" would close the loop.

A "shadow mode" deployment flag — audit and record but don't actually block — would dramatically reduce adoption friction in the first weeks at any org. Worth adding to TASK-1.10 as an effective-decision mode.

Evidence retention lifecycle (how long, what gets pruned, what's immutable for incident replay) is mentioned in TASK-1.15 but not specified anywhere as policy.

## Bottom line

The thesis is correct and the scope is honest. The two things most likely to bite are (1) cold-start being the weakest case for your strongest argument, and (2) compute/latency being unmodeled in a system whose UX promise depends on proactive auditing finishing before developers ask.
