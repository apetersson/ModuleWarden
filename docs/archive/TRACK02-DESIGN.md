## 2. THREE CONVERSATION FLOW DESIGNS

### Flow A — Supply-Chain Risk Query
> **Underwriter:** "What is the supply-chain risk of Acme GmbH?"

| Step | Actor | Action | Output |
|------|-------|--------|--------|
| 1 | **Chat UI** | Parse intent → route to `UnderwriterOrchestrator` | `intent: supply_chain_risk`, `entity: Acme GmbH` |
| 2 | **Orchestrator** | Spawn `Analyst` agent (Decepticon pattern) | Pull Acme GmbH SBOM / public repo list via OSINT |
| 3 | **Analyst** | Run dependency sweep | List of 147 npm deps, 3 flagged as high-risk (typosquat candidates, install-script present) |
| 4 | **ModuleWarden Gate** | Score each flagged package | `postmark-mcp@1.0.16 → score=0.94, decision=block`, `event-stream@3.3.6 → score=0.97, decision=block` |
| 5 | **Analyst** | Map to MITRE ATT&CK | T1195.001 (Supply Chain Compromise), T1567 (Exfiltration via Web Service) |
| 6 | **Orchestrator** | Synthesize NL + structured JSON | **NL:** "Acme GmbH runs 147 npm dependencies. Two packages scored above the block threshold: postmark-mcp (0.94) and event-stream (0.97). Both have install-time exfiltration patterns. This maps to MITRE T1195.001 and T1567." **JSON:** `{ "risk_score": 0.91, "blocked_packages": 2, "mitre_techniques": ["T1195.001", "T1567"], "recommended_premium_loading": "+18%" }` |

**How Decepticon generates it:** The Analyst agent uses the same dependency-CVE sweep logic as Decepticon's Analyst specialist. The Orchestrator uses Decepticon's OPPLAN middleware pattern (objective → agent → finding → status update). The NL response is generated by the Orchestrator LLM from structured findings, exactly as Decepticon writes FIND-NNN.md reports.

### Flow B — Control Evidence Memo Generation
> **Underwriter:** "Generate a control evidence memo for SOC 2 CC 7.1"

| Step | Actor | Action | Output |
|------|-------|--------|--------|
| 1 | **Chat UI** | Parse intent | `intent: evidence_memo`, `framework: SOC 2`, `control: CC 7.1` |
| 2 | **Orchestrator** | Spawn `Soundwave` planner clone | Interview underwriter for scope: "Which applicant? Which systems in scope? Testing window?" |
| 3 | **Soundwave** | Generate engagement bundle (abbreviated) | RoE (underwriter-owned), Threat Profile (MITRE-mapped adversary: "Supply-chain exfiltrator"), ConOps (kill chain: recon → initial access via dep install → exfil) |
| 4 | **Orchestrator** | Build OPPLAN from bundle | 3 objectives: (1) Enumerate applicant SBOM, (2) Simulate install-time payload, (3) Verify detection controls |
| 5 | **Recon + Exploit agents** | Execute OPPLAN objectives | Objective 1: SBOM extracted. Objective 2: Simulated malicious install script detected by ModuleWarden gate (blocked). Objective 3: SIEM alert fired within 45 seconds. |
| 6 | **Analyst** | Write structured memo | See Section 4 for the full template. |
| 7 | **Orchestrator** | Return NL summary + PDF artifact | **NL:** "The memo is ready. ModuleWarden blocked the simulated payload at install time. Your SIEM fired within 45 seconds. SOC 2 CC 7.1 is satisfied with evidence grade A." **Artifact:** `/memo/soc2-cc71-acme-20260529.pdf` |

**How Decepticon generates it:** This is literally Decepticon's engagement workflow, repurposed. Soundwave interviews → engagement bundle → OPPLAN → agent execution → findings. The only delta: the "target" is an applicant's SBOM, not an IP range, and the "attack" is a sandboxed simulation.

### Flow C — Simulated Supply-Chain Attack
> **Underwriter:** "Simulate a supply chain attack on this applicant's dependencies"

| Step | Actor | Action | Output |
|------|-------|--------|--------|
| 1 | **Chat UI** | Parse intent | `intent: simulate_attack`, `target_applicant: Acme GmbH` |
| 2 | **Orchestrator** | Confirm RoE with underwriter | "I will simulate a dependency-swap attack on the sandboxed replica. No production systems. OK?" |
| 3 | **Orchestrator** | Build OPPLAN: `SIM-001` | Objectives: (1) Recon applicant dependencies, (2) Build kill chain (T1195.001 → T1059 → T1041), (3) Execute in isolated sandbox, (4) Measure detection time |
| 4 | **Recon agent** | Objective 1 | Identifies `lodash` as high-dependency, low-entropy target for typosquat |
| 5 | **Exploit agent** | Objective 2 | Constructs `1odash` typosquat payload with benign postinstall beacon |
| 6 | **Post-Exploit agent** | Objective 3 | Runs payload in sandbox. ModuleWarden gate blocks at install. If bypassed, beacon reaches C2 (logged). |
| 7 | **Analyst** | Objective 4 | Measures: gate blocked in 3.2s, SIEM detected in 12s, no beacon if gate active |
| 8 | **Orchestrator** | Synthesize quantified risk | **NL:** "Simulation complete. The attack chain was blocked at the dependency gate in 3.2 seconds. Without the gate, the beacon would have exfiltrated within 12 seconds. Your estimated probability of loss drops from 0.34 to 0.02 with ModuleWarden active." **JSON:** `{ "sim_id": "SIM-001", "kill_chain": ["T1195.001", "T1059", "T1041"], "blocked_at_phase": "initial-access", "detection_time_ms": 3200, "p_loss_without_gate": 0.34, "p_loss_with_gate": 0.02, "premium_loading_delta": "-12pp" }` |

**How Decepticon generates it:** This uses Decepticon's full autonomous loop: OPPLAN → agent dispatch → execution → findings. The "sandbox" is the ModuleWarden audit container (Docker, no net, 1-shot). The kill chain maps directly to MITRE ATT&CK techniques. The quantified risk metrics are derived from simulation outcomes (detection time, block rate).

## 5. 36-HOUR BUILD PLAN (P0 / P1 / P2)

> **Philosophy:** A working Streamlit app with 3 hardcoded flows beats a fantasy full integration. The judges see a chat UI, real model output, and a PDF artifact.

### P0 — Hour 0-12: "It talks and it scores"
**Goal:** Working Streamlit chat + ModuleWarden gate inference + 1 conversation flow

- [ ] **Hour 0-2:** Scaffold Streamlit chat UI (`st.chat_message`, sidebar for applicant selection)
- [ ] **Hour 2-4:** Wire the existing `apiary-starter` gate: `POST /score` returns real CodeBERT/XGB scores
- [ ] **Hour 4-6:** Build Flow A end-to-end: underwriter asks supply-chain risk → UI calls gate for 3 hardcoded packages → renders NL summary + JSON panel
- [ ] **Hour 6-8:** Add "hardcoded Decepticon" layer: static OPPLAN JSON, static agent names, no real sandbox. The *illusion* of agent orchestration.
- [ ] **Hour 8-10:** Style the UI: UNIQA brand colors (blue/white), underwriter persona, print-friendly CSS
- [ ] **Hour 10-12:** Record 60-second demo GIF. Test on Leonardo A100 if model serving needs GPU.

**P0 Success Criteria:** A judge can type "What is the supply-chain risk of Acme GmbH?" and see a natural-language answer with real scores from the ModuleWarden gate.

---

### P1 — Hour 12-24: "It simulates and it memo-izes"
**Goal:** Flows B and C working + PDF generation + Neo4j graph visualization

- [ ] **Hour 12-14:** Build Flow B: static engagement bundle template → fill with applicant data → render in UI
- [ ] **Hour 14-16:** Build Flow C: hardcoded kill chain (T1195.001 → T1059 → T1041) → animate step-by-step in chat → show "BLOCKED at phase 1"
- [ ] **Hour 16-18:** Generate PDF evidence memo from template (use `weasyprint` or `fpdf2`). Download button in chat.
- [ ] **Hour 18-20:** Add Neo4j graph viz: static attack-chain graph rendered with `streamlit-agraph` or D3 iframe. One hardcoded graph per flow.
- [ ] **Hour 20-22:** Polish NL generation: prompt-engineer the Orchestrator responses so they sound like underwriters, not hackers. Remove "exploit", replace with "simulated adversarial action".
- [ ] **Hour 22-24:** End-to-end dry run. Fix latency (cache gate scores in Redis/SQLite).

**P1 Success Criteria:** Judge can run all 3 flows, download a PDF memo, and see a kill-chain graph.

---

### P2 — Hour 24-36: "It integrates and it wins"
**Goal:** Mock UNIQA product integrations + pitch polish + model training evidence

- [ ] **Hour 24-26:** Mock Policy Binding API: `POST /api/v1/bind` accepts applicant ID + SBOM → returns premium quote + evidence memo URL. Static response, realistic schema.
- [ ] **Hour 26-28:** Mock Renewal Risk Assessment: diff view showing "since last year, 12 new deps added, 1 blocked".
- [ ] **Hour 28-30:** Add "model card" page: show CodeBERT LoRA training curves from `apiary-starter/scripts/train_codebert.py`. Judges want to see "a real, trained model."
- [ ] **Hour 30-32:** Stress-test: run 50 concurrent chat sessions via `locust` or `pytest`. Screenshot latency metrics.
- [ ] **Hour 32-34:** Pitch deck: 5 slides. Problem → Solution (ModuleWarden + Decepticon) → Demo screenshot → Track 02 fit → Team.
- [ ] **Hour 34-36:** Final rehearsal. Deploy to Cloudflare Pages / Streamlit Cloud. Test on venue WiFi. Sleep 0 minutes.

**P2 Success Criteria:** The pitch says "We trained a model on Leonardo A100s, deployed a real gate, and built a conversational underwriter assistant that generates insurance-grade evidence from autonomous red-team simulations."
