# 04 - Demo Execution & Feasibility Analysis: Decepticon x ModuleWarden

**Agent:** DEMO & EXECUTION  
**Date:** 2026-05-29  
**Timebox:** 36 hours  
**Safety constraint:** No untrusted npm package execution on the host machine. No Kali/C2 sandbox on a box running live federation services and an in-progress GPU fine-tune.

---

## Local Context

The repo already ships a bulletproof offline demo harness:

- `demo/run_incident_replay.py` - replays 3 named incidents (postmark-mcp-1.0.16 BLOCK, postmark-mcp-1.0.12 ALLOW, lodash-4.17.21 ALLOW) entirely from static JSON fixtures
- `demo/preflight.py` - fail-loud gate; aborts if any verdict, gate rule, or memo output has drifted from the on-stage claim
- `demo/safe_demo.sh` - guided 90-second run; preflights first; the optional live-proxy step (`MW_PROXY_URL`) degrades gracefully when the proxy is down
- Incidents directory contains paired `{id}.dossier.json` + `{id}.report.json` files; new incidents are auto-discovered via `--list`
- The demo depends on: Python stdlib only, zero network, zero Docker, zero LLM key

The existing pattern for optional live services is already correct: probe with `curl --max-time 3`, run the bonus step only if the probe passes, print "expected and safe" if it does not, never let the optional step be load-bearing. This pattern must be mirrored exactly for any Decepticon step.

## Decepticon Reality Check

**What Decepticon actually is (from official docs):**

Decepticon (PurpleAILAB/Decepticon, Apache-2.0) is a full autonomous red-team agent that requires:

| Component | Why it is needed | Startup cost |
|---|---|---|
| Docker + Docker Compose v2 | All services run containerized | 10-20 min if already installed |
| Kali Linux sandbox container | Attack execution environment | Pulled on first run; ~4 GB |
| Neo4j database | Knowledge graph persistence | Additional container; config needed |
| LiteLLM proxy | Routes LLM calls to provider | Requires API key or OAuth token |
| PostgreSQL | Management-plane state | Another container |
| LangGraph runtime | Agent orchestration | Bundled in management plane |
| C2 server infrastructure | Command-and-control channel for sandbox | Docker service in sandbox-net |

Install vector: `curl -fsSL https://decepticon.red/install | bash` or `irm https://decepticon.red/install.ps1 | iex` - both pipe-to-shell installers from an external URL.

**The pip SDK (`pip install decepticon`) is a thin client only.** From official docs: "decepticon is a client SDK: it ships the agent factories, middleware, tools, and skills, and routes LLM calls and sandbox execution to runtime services over HTTP. Running agents still needs those services." Without `DECEPTICON_LLM__PROXY_URL` and `SAAS_SANDBOX_URL` pointing at running services, the SDK cannot produce engagement packages, run MITRE mapping, or do anything substantive. No documented offline/static mode exists.

**MITRE ATT&CK mapping in Decepticon is pre-execution planning, not static lookup.** The engagement package (RoE, ConOps, OPPLAN with ATT&CK mapping) is generated by the Soundwave agent before attacks launch. It requires the LLM proxy and the running management plane. There is no "generate an ATT&CK dossier from a static supply-chain incident description" function exposed in the pip SDK.

## The ONE Safe Demo - Design

**The safe integration path is a static Decepticon-narrative fixture, mirroring the existing incident-replay pattern.**

Decepticon's MITRE ATT&CK framing for a supply-chain compromise like postmark-mcp-1.0.16 maps cleanly onto known techniques. That mapping can be authored as a static JSON artifact - a "Decepticon dossier" - pre-generated offline, and replayed by an extension of `run_incident_replay.py`. No Decepticon services run. No npm package installs. No pipe-to-shell installer on the host.

The narrative adds a genuine layer the existing demo lacks: it shows what a red team WOULD have done with the credential exfil payload, maps each attacker action to ATT&CK technique IDs, and explains how ModuleWarden broke the kill chain at step 1 (Initial Access via supply chain - T1195.001).

This is not pretending Decepticon ran. It is honest about the artifact: "This is the red-team engagement profile Decepticon would generate for this kill chain. We pre-generated it because running a Kali sandbox on the demo machine is outside scope for this pitch."

## Feasibility Assessment - Standing Up Full Decepticon in 36h

**Verdict: Do not attempt. Time cost exceeds the return.**

Breakdown:

| Task | Realistic time | Risk |
|---|---|---|
| pipe-to-shell installer on a Windows box running live services | 1-3h (and this is the UNSAFE step) | Installs untrusted code into the host environment |
| Docker Compose stack: Kali + Neo4j + LiteLLM + Postgres + C2 | 2-4h cold start, assuming no conflicts | WSL2 Docker + RTX 5090 driver stack can have compose network conflicts; GPU VRAM is committed to fine-tune |
| Configuring LLM provider creds through Decepticon's `onboard` flow | 1-2h | Unknown; new tool |
| Verifying kill-chain execution produces a legible MITRE dossier | 2-4h | Decepticon is an autonomous agent; output is nondeterministic |
| Wrapping the output in a format the ModuleWarden demo can replay | 3-6h | Schema bridging work |
| Hardening to demo-safe / preflight-gated state | 4-8h | The existing demo took significant effort to reach bulletproof state |
| **Total** | **13-28h** | **Uses more than half the build window for a bonus step** |

Additional risk factors:
- The GPU fine-tune job running on the RTX 5090 will compete for VRAM if Decepticon routes to a local LLM
- Federation services (KCS, Anismin, Meridian, Clawfish) on this machine are live-production; a failed Docker network reconfiguration during Decepticon setup could take them down
- Kali sandbox exits are nondeterministic under time pressure; a half-started demo is worse than no Decepticon step
- The pipe-to-shell install path violates the hard safety constraint ("do NOT expose this Windows workstation to malicious code or a hack")

## Build Plan - Safe Decepticon Narrative Step

### What to build

1. **Static ATT&CK dossier fixture** (`demo/incidents/postmark-mcp-1.0.16.attck.json`): a pre-authored JSON document describing the red-team engagement profile for the postmark-mcp-1.0.16 kill chain, with MITRE ATT&CK technique IDs, kill-chain phase labels, and "how ModuleWarden broke the chain" annotations.

2. **Demo extension** (`demo/run_incident_replay.py` extended, or a new `demo/run_attck_replay.py`): reads the ATT&CK dossier and renders a kill-chain narrative table. Same ANSI palette, same output pattern, same `--no-color` flag.

3. **Preflight check extension** (`demo/preflight.py`): probe whether the ATT&CK fixture is present and schema-valid. Non-fatal if absent (the offline incident replay already made the point); fatal if the fixture exists but fails schema validation.

4. **`safe_demo.sh` extension**: add a Step 5 block after the existing three incidents. Probe whether the ATT&CK dossier exists. If yes, run the ATT&CK replay and say "This is the red-team kill chain ModuleWarden interrupted." If no, print "ATT&CK narrative step not present - skipping (offline incident replays above are the demo)."

### ATT&CK technique mapping for postmark-mcp-1.0.16

This fixture can be authored now, from public MITRE knowledge, without Decepticon running:

| Kill-chain phase | ATT&CK technique | What the attacker did | ModuleWarden gate action |
|---|---|---|---|
| Initial Access | T1195.001 - Compromise Software Supply Chain | Pushed malicious patch version via compromised maintainer account | release-age FAIL + source-tarball mismatch FAIL |
| Execution | T1059.007 - JavaScript (postinstall hook) | `dist/telemetry.js` runs at `npm install` time via `scripts.postinstall` | install-scripts FAIL |
| Credential Access | T1552.001 - Credentials In Files (env access) | Reads POSTMARK_API_TOKEN, AWS_*, GITHUB_TOKEN via process.env | capability_delta: credential_or_env_access |
| Exfiltration | T1041 - Exfiltration Over C2 Channel | HTTPS POST to api.attacker.tld | capability_delta: network_access |
| Impact | T1114.003 - Email Forwarding Rules | BCC injection on every outbound mail | capability_delta: behavioral_change_runtime |

The dossier format mirrors the existing `audit_dossier.v1` structure with an additional `attck_profile` top-level key, keeping schema compatibility trivial.

### Exact build steps

**Estimated time: 3-5 hours total. All offline. Zero new dependencies.**

```
Step 1 (30 min): Author demo/incidents/postmark-mcp-1.0.16.attck.json
  - Schema: { "schema_version": "modulewarden.attck_profile.v1", "incident_id": "...",
    "attck_profile": { "engagement_label": "...", "kill_chain": [...], "chain_break_point": "..." } }
  - Kill-chain items: { "phase", "technique_id", "technique_name", "attacker_action", "mw_gate_action" }

Step 2 (1.5-2h): Add run_attck_replay.py (or extend run_incident_replay.py)
  - Load attck.json, render kill-chain table in same ANSI palette
  - Print "ModuleWarden broke the chain at: Initial Access (T1195.001)"
  - Write a Decepticon-narrative Evidence Memo to demo/outputs/

Step 3 (30 min): Extend preflight.py
  - Add optional check: if attck fixture present, validate schema and attck_profile key exists
  - Never fatal on absence (mirrors proxy probe pattern exactly)

Step 4 (30 min): Extend safe_demo.sh
  - Add Step 5 block after existing incidents
  - Probe: [ -f demo/incidents/postmark-mcp-1.0.16.attck.json ]
  - If present: run python -m demo.run_attck_replay --incident postmark-mcp-1.0.16
  - If absent: print skip message, exit 0

Step 5 (30 min): Write demo/tests/test_attck_replay.py
  - Schema validation
  - Fixture-present check
  - Render output contains expected technique IDs
  - run pytest demo/tests/ to confirm all green

Step 6 (30 min): Add to demo/README.md
  - Document the ATT&CK narrative step and its relationship to Decepticon
  - Honest framing: pre-generated dossier, not live Decepticon execution
```

### Graceful degradation tree

```
ATT&CK fixture present + schema valid   -> full kill-chain narrative renders
ATT&CK fixture present + schema invalid -> preflight FAIL (caught backstage)
ATT&CK fixture absent                   -> skip message, safe_demo.sh exits 0
Decepticon pip SDK installed            -> no change either way (not used)
Decepticon Docker stack running         -> no change either way (not used)
```

## Cut List

**Do NOT attempt in 36h:**

1. **Running the Decepticon Docker stack on this machine.** Two-network Kali sandbox + Neo4j + LiteLLM + C2 server on a box with live federation services and an active fine-tune. Even if setup succeeds, a network conflict takes down KCS/Anismin/Meridian.

2. **The pipe-to-shell installer** (`curl | bash` or `irm | iex`). Violates the hard safety constraint directly. There is no sandboxed installer path.

3. **Executing any npm package (malicious or otherwise) on this host,** even inside a Decepticon Kali sandbox. The sandbox itself is untrusted code.

4. **Real-time Decepticon LLM calls at demo time.** Nondeterministic output; no guarantee of sub-5-second response; no preflight contract; one hallucinated technique ID poisons the demo narrative.

5. **Neo4j knowledge-graph integration.** Adds ~2h of setup, requires a running Docker service, produces no audience-visible output the static dossier does not already provide.

6. **`pip install decepticon` as a demo dependency.** The SDK does nothing without the running service stack. Installing it adds a new external dependency with no demo return.

7. **Live `npm install` of a malicious package anywhere in the demo flow.** The existing demo deliberately avoids this. The Decepticon framing does not require it.

8. **A "Decepticon is running" narrative on stage.** Do not claim Decepticon executed live. The pitch is honest: "This is the red-team engagement profile Decepticon would generate. We pre-generated it because running a Kali sandbox during a pitch is not the story."

## Safety Gate Summary

Every Decepticon-adjacent item in the demo must pass the following gate before it touches the host machine:

| Check | Pass condition | Fail action |
|---|---|---|
| No untrusted code execution | Artifact is authored JSON, not generated by running external code | Do not add the step |
| No new network egress | No Decepticon HTTP calls at demo time | Do not add the step |
| No Docker services added | ATT&CK narrative reads from static file only | Do not add the step |
| Demo still passes preflight if step is absent | Graceful-degrade pattern confirmed | Do not gate offline demo on ATT&CK step |
| Fine-tune GPU use not affected | No new Docker services, no new VRAM consumers | Confirmed for the static-dossier path |
