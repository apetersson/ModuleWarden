motivation: popular package maintainer has a security problem. what to expect?
a sophisticated attacker will craft a not statically detectable exploit.

-----------
probabilistic risk assesment.
graph dependency complexity vs compounded risk.
------------

HOW TO DEFEND?
we guard the interface between code and developer: the artifact registry.
the service where you get your packages from.
defend: agentic audit with modulewarden.

one code review you start manually is good.
twenty that run automatically on every new version don't get tired.

-----------
the pipeline:
1. new version lands → enqueued for review
2. evidence extraction runs:
   - static checks (exec calls, obfuscation, network/fs/process capabilities)
   - advisory database lookup
   - dependency diff against previous version
   - capability delta report
   - git history → monthly commit/contributor/code_quality time series
   - npm downloads → 5-year monthly buckets
3. temporal forecasting (Sybilion):
   - submit time series per metric to Sybilion API
   - poll for forecast.json artifact
   - extract collapse_risk, uncertainty, recent_anomaly per metric
   - composite temporal_risk score (0-1) from weighted aggregation
   - store as temporal_evidence.json in evidence artifacts
4. prompt packs assemble audit instructions from configured markdown files
5. Docker container spawns with all evidence + instructions
6. model reads evidence, returns structured verdict (ALLOW/BLOCK/QUARANTINE)
   with every claim citing its evidence source
7. on ALLOW → promote tarball to Verdaccio registry

-----------
temporal forecast dashboard card (IMPLEMENTED):
- per-metric rows: commits, contributors, code_quality, downloads
- collapse risk badge, uncertainty score, anomaly flag per metric
- linked Sybilion job ID for each forecast
- composite temporal_risk badge (0-100%)
- visible in every audit run detail

-----------
the model:
- 27B Qwen variant, fine-tuned on 1,800 CVE codebases
- trained on Leonardo supercomputer
- two LoRA adapters, published on Hugging Face
- works with any OpenAI-compatible /v1 endpoint
- held-out val loss 0.21 (measures narration fidelity, not detection accuracy)

-----------
the prompt packs:
- markdown instruction files, human-readable and version-controlled
- core packs: claimed-purpose-vs-behavior, install-time-execution, advisory review, version-diff
- pattern packs: crypto-web3-hijack, secret-harvesting, protestware, remote-payload
- custom packs written by security teams for their threat model
- packs are assembled with the evidence bundle and sent to the model

-----------
the verdict:
- ALLOW → promote the tarball into the Verdaccio registry → developers can install
- BLOCK → reject at the proxy → `npm install` returns 403
- QUARANTINE → hold for manual review → visible in the dashboard column

-----------
honesty beats:
- Sybilion forecast cannot distinguish dying from healthy packages (AUROC 0.54)
- the model val loss (0.21) measures how well it writes up evidence, not how well it detects
- every verdict cites its source evidence lines
- Sybilion job IDs are preserved so you can audit the forecast yourself
